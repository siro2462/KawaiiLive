# roadmap-next — ロードマップの次ステップを実装する

## 手順

1. **現在地の確認**
   - `docs/ロードマップ.html` の「依存関係と優先順位（2026-07-02 引き直し）」→「最短ロードマップ（テーブル版）」を読み、状態が ⬜ の最初の行を特定する
   - CLAUDE.md の「現在地」も突き合わせる（食い違ったらロードマップが正）

2. **仕様の読み込み**
   - 特定したタスクに対応する詳細セクションを読む:
     - A1〜A3 / B4〜B6 / C7〜C9 → 「ベクトルDB活用・雑談品質改善」の Phase A/B/C 表（変更モジュール・具体的変更が書いてある）
     - D-1〜D-4 → 「Phase D」の各実装詳細（スキーマ・プロンプト・パラメータ表まである）
     - E系 → 「Phase E」
   - **仕様に書いてあることをそのまま実装する**。設計をやり直さない

3. **実装の原則**
   - 最小変更。別形式・別システム・新しい抽象層を作らない（ロードマップの削減表で「作らない」と決めたものを復活させない）
   - 対象ファイルは仕様の「変更モジュール」列にあるものだけ。周辺のリファクタはしない
   - プロンプト文言はロードマップに書いてあるテンプレをベースにする（勝手に要約・改変しない）

4. **検証**
   - JS変更: `node --check <file>` で構文確認
   - UI変更: `cd app/ui && npx vite build`（必須。distを直配信している）
   - server.js / app配下の.js変更はサーバー再起動が必要な旨をユーザーに伝える
   - D-4案1（分布メトリクス）が実装済みなら、生成テストして before/after を出す

5. **完了処理**
   - `docs/ロードマップ.html` の該当行の状態を ⬜ → ✅ に更新し、必要ならチェックリストの `class="todo"` → `class="done"` も更新
   - 何を変えたか・検証結果・次のステップを報告する

## 注意

- Step 0 の3項目（A1+A2+A3 / D-2案1 / D-4案1）は独立なので、ユーザーが指定しなければ番号順に1つずつ
- D-2案2（コメント割り込み）は「初コメントが付いたら」の状況トリガー。順番を無視して最優先になる
- コミットはユーザーが指示した時だけ。メッセージは日本語
