memory の追加手順

ステップ内容
1. ネタ出しネタ元は3系統。(a) 日常の出来事を ZIPちゃん設定に翻案（実在人物は出さない。チェーン店・商品名はOK、布教の熱量になる） (b) ターゲット台本のトピック構造の翻案（内容コピーはNG、「布教」「企画振り返り」等の型だけ借りる） (c) 季節・イベント（正月、花見、新作ゲーム）。1回あたり10〜20件を目安にネタの種を作る。種は「状況+具体物」まで書くと安定（「コンビニの話」ではなく「雨上がりの夜にコンビニでビニール傘を2本目買った」）
2. LLM整形下の「memory生成プロンプト」に種を1件ずつ渡し、B6の新カラムまで一度に埋めたJSONを得る。episode（事実文）と episode_v2（話し言葉）を同時生成するので後工程が不要
3. 投入JSON配列を data/memory-source/YYYYMMDD.json に保存し、scripts/add-memories.mjs（新設・append専用）で挿入。replaceTalkMemoriesFromJsonFiles は全置換のため使わない（廃止対象）
4. 検証投入後に keywords重複チェック（既存とのコサイン類似 or キーワード一致で疑似重複を警告）。既存455件へのバッチ適用も同じプロンプトでできる: episode を種として渡し、習慣形（「〜することがある」）を一回きりの断片記憶に書き直し + episode_v2 未生成分（112件）+ 新カラム後付けを一度に処理する

memory生成プロンプト（テンプレ） — 種1件 → memory 1件。
ポイント: JSONを一発で書かせると習慣形の一般論（「〜することがある」）に寄る。先に「一回きりの出来事」のディテールを固めさせてから episode を書かせる。

あなたは配信者「ZIPちゃん」の記憶を作る。目的は、本人しか知らない一回きりの断片記憶。
実在の人物・配信者の出来事は使わない。店名・商品名・値段は実在のものを使ってよい。

ネタの種: {{seed}}    （例: コンビニで新作スイーツを買って冷蔵庫で忘れた）

手順（1〜4を考えてから、最後にJSONだけを出力する）:
1. この出来事が起きた「一回」を決める: 曜日・時間帯・天気・直前に何をしていたか
2. 断片ディテールを8個列挙する: 値段/個数/棚の位置/音/匂い/手の動き/他の客/店員の様子
3. そのうち「本筋と関係ないのに妙に覚えていること」を1つ選ぶ
4. 「思い出せない部分」を1つ決める（なんでそうしたのか・その後どうしたか等）

出力形式:
{
  "keywords": ["…", "…"],            // 検索用 2〜4語
  "episode": "…",                     // 一回きりの出来事として3〜5文。数字(値段/時刻/個数)を
                                      //   最低1つ、五感の断片を最低1つ入れる
  "episode_v2": "…",                  // 同じ内容をZIPちゃんの一人称の話し言葉で3〜5文
  "category": "food|game|fail|discovery|outing|goods|health|season",
  "mood": "布教|笑い話|しみじみ|グチ|発見",
  "people": ["店員さん"],             // 登場人物。いなければ []
  "punchline": "…",                   // この話のオチ・一番おいしい所を1文
  "objects": ["…"],                   // 位置・状態つきの具体物 2〜4個（レジ横の棚に2個だけ残ってたプリン）
  "actions": ["…"],                   // 小さい行動 2〜3個（値札を二度見した、袋を持ち直した）
  "micro_conflicts": ["…"],           // 小さい矛盾・違和感 0〜2個（買ったのに食べてない）
  "odd_detail": "…",                  // 本筋と関係ないのに覚えていること1つ（手順3）
  "unclear": "…"                      // 思い出せない部分1つ（手順4）
}

条件:
- 「〜することがある」「よく〜する」の習慣形は禁止。その日一回の出来事として書く
- 抽象語（感覚・違和感・空気感・現実感）で説明しない。物と行動と数字で書く
- きれいなオチをつけすぎない。人の記憶は途中で終わってよい
- episode_v2 は「〜なんだよね」「〜しちゃってさ」等の配信口調。顔文字・記号は使わない

悪い例（習慣形・抽象的）:
「コンビニで新作スイーツを買った。冷蔵庫に入れて忘れることがある。夜中に思い出して食べた。」

良い例（断片記憶）:
「火曜の23時ごろ、雨上がりにファミマに寄った。新作の生カスタードプリンがレジ横の棚に
2個だけ残ってて、なぜか2個とも買った。前の客がホットスナックを3種類買ってて、袋に
揚げ物の匂いが移った気がした。1個はその日に食べて、もう1個は冷蔵庫の奥で金曜まで
忘れてた。なんで2個買ったのかは覚えてない。」

分類カラム（category/mood/punchline/people）は B6、atoms（objects/actions/micro_conflicts）は Phase 3 の先取り。odd_detail は生成側の UNNECESSARY_DETAIL angle、unclear は lost_thread / self_question の素材にそのまま使える。カラムが未追加の間は無視されるだけなので、データ側を先にこの形式へ揃えてよい。

2 Xトレンド X API は有料化済みでコスパが悪い。Google Trends で代替し、X は「面白いのを見かけたら手動で種として投入」に留める API契約はしない（明確に非推奨）
3 季節カレンダー（静的） 祝日・季節行事・注目ゲーム発売日を data/calendar.json に事前記入。API 不要で確実。E-2 のイベント生成の入力を兼ねる 今すぐ作れて外部依存ゼロ。鮮度はないが「季節の話」は雑談の定番

推奨: 案3（静的・今すぐ）→ 案1（日次RSS）。案2はやらない。

E-4: 反応学習（ウケの還流）

C7 の section/topic_title 単位で「何がウケたか」を計測し、3箇所へ還流する。同接が少ない間はコメント1件を強いシグナルとして扱う（データが薄い前提の設計）。

還流先方法
① ネタ選定の重みmemory に response_score カラム追加。コメントが付いたトピックの category / canon を planner で加点。翌週の台本に「ウケた系統」が増える
② style 還流コメントが付いた直前の言い回しを style カード候補として自動抽出 → 月次で人間が選別してから DB 追加（自動追加はしない）
③ 常連記憶viewer テーブル新設: {handle, first_seen, last_seen, comment_count, topic_tags}。「◯◯さん、いつものプリンの人だ」が言える。保存はハンドル名と話題タグのみ（コメント全文の長期保存はしない）

E-5: 成長の可視化（バズ要素としての成長）

  • 配信回数・記念回・「初めてできた」を canon (kind: milestone) に記録し、台本で自己言及（「この配信も30回目らしい。びっくりだね」）
  • canon.mention_count から「よく話す話題ランキング」を自己ネタ化（「私プリンの話しすぎらしい」）— 統計がそのまま人格ネタになる
  • 外側ループ接続: 配信後に「切り抜きどころ」を自動抽出（コメント密度 + LLM判定）して既存のショートパイプラインへ渡す → 切り抜き→流入→コメント増→ E-4 の学習が加速

安全ゲート（Phase E 共通）

  • 全ての自動書き込み（memory / canon / news_card）は承認キュー経由から開始。ダッシュボードに「今日の経験（承認待ち）」パネルを追加
  • 自動化解禁の基準: 承認キューの却下率が2週連続 20% 未満になったソースから順に
  • 経験の総量制限: memory 追加は日次 max 5件。古く response_score の低い memory は retired（無限に増やすと検索が濁る）
  • 実在視聴者: ハンドル名のみ・ネガティブな記憶は作らない・晒し/いじりはしない
  • ニュース: news_card の範囲外を語らない。訂正報道が出たら expires
  • ファインチューニングによる「学習」はやらない（経験= データループで実現。FT は memory が数千件を超えプロンプト注入で個性が出なくなってからの最終手段）

結論

台本を作る機能は育ってきた。次は「つなぎ込み」と「生きてる感」。新しい仕組みはもう増やさない。

queue・独自スコア体系・全面スキーマ統一のような「新しい形式づくり」は全て既存テーブルの拡張に縮小した（詳細は「依存関係と優先順位」の削減表）。

今日入れられる3つ（Step 0）:
  1. A1〜A3: ベクトルDBをプロンプトに実際に届かせる
  2. D-2案1: 存在しないコメントを読む演出をやめ、独り言スタイルに統一する
  3. D-4案1: ターゲット台本との分布メトリクスを基準値ごと用意する（以後の舵取り装置）

その後は C7（番組構成）→ D-1 / D-3（canonと音声表現）が本丸。
初コメントが付いたら D-2案2（リアルコメント割り込み）を全てに優先して前倒す。

最終形は Phase E（経験を積む配信者）: 配信・コメント・外的要因から日々 memory / canon が育ち、配信するほど人間らしくなるループ。バズは (a)切り抜き導線 × (b)成長物語 × (c)常連コミュニティ の複利で作る。
