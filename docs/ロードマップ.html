TODO Phase 1: 10本生成→1本選定システム

毎日10本作るなら、自動評価が必須。全部人間が見るのは無理。

評価スコア体系

quality_score
  ├─ punctuation_score       句読点バグ
  ├─ leak_score              内部ラベル漏れ
  ├─ abstract_score          抽象感覚語の数
  ├─ duplicate_score         重複率
  ├─ kaidan_score            怪談パートの品質
  ├─ first_live_score        初Live適合度
  ├─ naturalness_score       自然さ
  └─ broadcast_safety_score  配信安全性

自動で落とす条件

  • 内部ラベル漏れあり
  • 句読点バグ多い
  • KAIDAN_MAINが900字超え
  • 前回/今日も/おかえり等が初Liveに混入
  • 怪談がレポート化
  • previous_tailコピーが強い
  • 制御語 ルール プロンプト 本文にない が本文に出る

完了形

10本生成 → 自動評価 → 上位3本だけ表示 → ChatGPT/人間が最終確認 → 採用1本を配信用queueへ変換

TODO Phase 2: 怪談DBの整理

怪談 raw text をそのまま渡さず、stream_card に変換する。

stream_card 形式

{
  "title": "七人みさき",
  "one_line": "七人一組で現れるとされる伝承。",
  "key_points": [
    "七人の数が変わらないと語られる",
    "出会った人が次の一人になるという話がある",
    "海や川で亡くなった人の霊として語られることがある"
  ],
  "stream_angle": "人数がずっと七人のままという決まりごとの嫌さ",
  "daily_return_hooks": ["夜道", "人数を数える", "駅", "コンビニ帰り"],
  "do_not_say": [
    "事実だと断定しない",
    "現地で体験したように話さない",
    "資料説明を長くしない"
  ],
  "risk_level": 2,
  "scary_level": 3
}

怪談DBに追加したい情報

フィールド用途
summary_for_stream配信用の短い要約
key_points核心ポイント（最大3つ）
stream_angle配信で話す切り口
return_hooks日常に戻すhook
risk_level配信リスク
scary_level怖さレベル
used_count使用回数
last_used_at最終使用日
  • generatorにraw_textを直接渡さない
  • 怪談の説明が最大3文で済む
  • 日常雑談に戻るhookがある
  • 同じ怪談を連続使用しない
  • リスク高い怪談を避けられる

特に重要な追加フィールド: actions, micro_conflicts, social_gaze
これを入れると、抽象語に逃げにくくなる。

topic DB → topic_pattern として再定義

topicは具体素材ではなく、話の運び方として使う。

{
  "handling": "小さい恥ずかしさから、他人の目線へ進め、最後に自己ツッコミで落とす",
  "preferred_atom_types": ["micro_conflict", "social_gaze", "action"],
  "avoid": ["abstract_explanation", "lesson_like_summary"]
}

注意: topic_hint を自由文で復活させるのはNG。また別話題混入する。

style DB → style_moves へ変換

{
  "move_type": "self_tsukkomi",
  "trigger": "小さい行動をした直後",
  "operation": "自分で軽くツッコむ",
  "example": "いや、何してんの私。"

ACTIVE Phase 4: 配信用queueとランタイム

生成CSVをそのまま配信に流すのは危ない。中間形式が必要。

broadcast queue 項目

{
  "id": 1,
  "text": "あー、聞こえてるかな。",
  "tts_text": "あー、聞こえてるかな。",
  "subtitle_text": "あー、聞こえてるかな。",
  "section": "OP",
  "emotion": "neutral",
  "motion": "idle",
  "can_interrupt_after": true,
  "comment_reply_mode": "short",
  "bgm_mode": "normal",
  "risk_flags": []
}

ランタイムに必要なもの

  • TTS queue (app/onair/radio.js)
  • 字幕表示 (SubtitleOverlay)
  • キャラ表示 (ClipAvatar)
  • 待機画面
  • 本編画面
  • ED画面
  • OBS制御 (接続済み、シーン切替は手動)
  • エラー時のつなぎ台詞
  • 手動停止ボタン

結論

台本を作る機能は育ってきた。でも、素材管理・選定・配信実行の土台がまだ弱い。

次にやるべきは生成モデルをいじることではなく、DB整理 + 評価 + 配信queue を作ること。

特に重要な3つ:
  1. 怪談DBを stream_card 化する
  2. memoryを atoms 化する
  3. 10本から選ぶ自動評価を作る

ここができると、毎日1本の安定配信にかなり近づく。
