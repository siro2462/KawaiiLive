KawaiiLive データフロー解説

SQLite / vector-db / audio / logs / UI / 最終更新: 2026-06-26

全体フロー

memory + vector-db + kaidan research
  → app/daihon/generate.mjs
  → script items
  → app/onair/script.js
  → data/talk-items.sqlite/live + speech_lines
  → app/onair/tts.js
  → data/audio/live-{id}/*.wav
  → app/onair/radio.js
  → playback + subtitle state
  → app/ui polling /api/status, /api/lives, /api/live/lines

主要データ

データ場所用途
memorydata/talk-items.sqlite台本生成の素材。id, keywords, episode を主に使う。
livedata/talk-items.sqlite1本の配信台本単位。タイトル、日付、状態、現在位置、評価値を持つ。
speech_linesdata/talk-items.sqlite台本の各ブロック/発話。本文、memory、anchor、LLM req/res、遷移情報、音声 path、再生状態を持つ。
vector DBdata/vector-db/{style,topic,flow}.sqlite話し方、話題、流れの検索。app/daihon/vector.mjs が読む。
audiodata/audio/live-{id}/Irodori が生成した wav。speech line ごとに単体または chunk 配列で保存。
script plan mirrordata/logs/script-plan.json最新 plan の確認用 mirror。実体は DB。
LLM logdata/logs/script-llm-v3.jsonlv3 の planning/generation/quality に関するログ。
radio statedata/logs/radio-state.json再生履歴と runtime state。version は 3。

台本生成時

  1. UI が POST /api/script/prepare を呼ぶ。
  2. app/server.js が model profile を解決し、RadioRuntime.prepareScript() を呼ぶ。
  3. app/onair/script.js が分数から block 数を計算し、app/daihon/generate.mjs を呼ぶ。
  4. generate.mjs が memory、vector、怪談 research を使って script item を作る。
  5. app/onair/script.js が live と speech_lines へ保存し、script-plan.json を更新する。

音声合成時

  1. UI が POST /api/live/synthesize を呼ぶ。
  2. RadioRuntime.synthesizeAudio() が対象 live の未合成 speech line を取得する。
  3. splitSpeechText() が長文を約200文字の chunk に分ける。
  4. Irodori worker が wav を生成し、speech_lines.audio_path に JSON 配列として保存する。

再生時

  1. UI が POST /api/radio/start を呼ぶ。
  2. RadioRuntime.start() が指定 live または最新 DB plan を読む。
  3. 音声が欠けていれば開始せずエラーにする。
  4. 音声 path を queue に積み、PowerShell の Media.SoundPlayer で同期再生する。
  5. 再生済み sequence は live.current_sequence_no と speech_lines.status/spoken_at に反映される。

UI から見えるデータ

app/ui/src/api.ts は /api/lives、/api/live/lines、/api/sqlite、/api/vector/table、/api/status を中心にポーリング/取得する。UI は DB と runtime のビューであり、直接ファイルを書き換えない。
